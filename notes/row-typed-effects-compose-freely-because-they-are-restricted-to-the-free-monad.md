# row-typed effects compose freely because they are restricted to the free monad

Leijen's key distinction from general monadic programming: algebraic effects are a *restriction* on monads -- specifically, to the free monad generated by the effect's algebraic signature. This restriction purchases composability for free.

With monad transformers (Haskell's mtl), composing `State` and `Exception` requires implementing `StateT` over `ExceptT` or vice versa, and the composition is not commutative -- different orderings give different semantics. Each new monad combination requires new implementation work.

With algebraic effects, composition is automatic: effects are rows of labels, and a function with effects `⟨state, exc | e⟩` simply means "both state and exception operations are available, plus any others." The type system tracks what effects a computation has; handlers discharge specific effects from the row. Composition is structural.

The non-commutativity of monad composition resurfaces, but constructively: `state(0, amb(surprising))` versus `amb(state(0, surprising))` give different results (shared vs. local state per branch), but both are valid and the different semantics are explicit from the handler nesting order. The programmer controls the semantics through handler ordering rather than discovering emergent behavior from monad stack order.

Leijen demonstrates this with the `surprising` function combining `state` and `ambiguity`: outer state gives global state across all ambiguous branches, inner state gives independent state per branch. This is a feature, not a bug -- the type system tracks both; the programmer chooses.

Relevance to [[language-design]]: if murail's composition language (Stage 9) ever adopts algebraic effects for resource effects (audio context, timing, scheduling), composability is the primary benefit. Multiple effects on a single computation don't require a custom monad stack -- they are just rows. Contrasts with [[evolvability-requires-trading-provability-for-extensibility]]: Sussman buys composability through runtime generic dispatch; algebraic effects buy composability through type-level row extension, preserving static checking.

The composition non-commutativity surfaces constructively: [[multi-effect-composition-order-determines-state-scope-in-algebraic-effects]] shows that handler nesting order gives explicit programmer control over whether state is shared or local across branches -- the semantics is determined by nesting order, not hidden in the monad stack definition.

The free-monad restriction is what makes [[handling-a-computation-is-composing-it-with-the-unique-free-model-homomorphism]] possible: the unique homomorphism theorem requires the free model to be a bona fide free algebra, which holds only for algebraic (not all monadic) effects. [[continuations-are-the-only-standard-computational-effect-that-cannot-be-represented-algebraically]] marks the boundary of where this composability holds.
