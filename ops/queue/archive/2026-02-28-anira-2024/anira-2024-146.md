# Task: onnx runtime does not support stateful lstm operations restricting it to stateless neural architectures

## Source Context

From [[anira-2024]] (Ackva & Schulz, TU Berlin, arXiv:2506.12665v1):

ONNX Runtime is architecturally stateless by design: it does not retain state between inference calls. This is documented in the ONNX Runtime architecture reference (cited as [28]). The consequence for audio neural networks:

- **Stateful LSTM (RNN-2k):** requires the LSTM internal state to persist and update across consecutive buffers (each buffer processes one sample, the next buffer continues from where the last left off). This state-carries-over requirement is incompatible with ONNX Runtime's stateless model.
- **Stateless LSTM (HNN-11k hybrid):** uses a variant of LSTM where state does NOT carry over between batches. This is compatible with ONNX Runtime and appears in both DS-I and DS-III datasets.
- Workaround noted by authors: state could be managed externally (passed in/out as tensors per inference call), but this alters the model graph structure. Excluded from the study.

This architectural constraint is not a performance limitation but a fundamental capability boundary: certain classes of sequential audio models cannot use ONNX Runtime as-is. LibTorch and TensorFlow Lite support stateful operations natively.
