# Task: larger model input sizes improve per-sample inference performance making latency tolerance a throughput lever

## Source Context

From [[anira-2024]] (Ackva & Schulz, TU Berlin, arXiv:2506.12665v1):

Statistical finding from LMM-I buffer size analysis:

Pairwise comparisons of buffer size predictor: almost all adjacent buffer size pairs show highly significant RpS reduction with larger buffers (p < 0.0001), consistent across all three engines. Effect is most pronounced at low buffer sizes (64-512 samples) and becomes less pronounced at larger sizes.

Authors' conclusion: "for applications with less stringent latency requirements, the use of a larger model input size is advisable."

The connection to latency: in anira's architecture, the model input size is fixed at compile time (InferenceConfig.model_input_shape). When model input size exceeds host buffer size, anira must buffer host audio until enough samples accumulate — adding latency (H_adapt in the latency formula). But if latency is acceptable, using a larger model input processes more samples per inference call, improving amortized per-sample cost.

This is a design lever: latency budget determines what model input sizes are feasible. A real-time audio effect plugin with 10ms latency tolerance can use a much larger model input than a plugin requiring sub-millisecond latency.

Note: the benchmarks in this paper aligned model input size with buffer size (no mismatch) specifically to isolate this variable. Real-world mismatched configurations remain unstudied.
